# System Architecture

## Overview

zk-proof-of-origin is a two-layer system:

1. **Circuit layer** — an o1js zkProgram that produces a single recursive proof asserting content origin (human or AI).
2. **Application layer** — a browser-based tool for creators to generate and share proofs, and for viewers to verify them.

## Circuit Design

### Core zkProgram: `OriginProof`

The zkProgram takes a content hash and an origin assertion, producing a succinct proof.

```
OriginProof(contentHash, originAssertion) → Proof
```

**Phase 1 (Human Provenance):**
```
contentHash:       Poseidon hash of the content (text, image bytes, etc.)
originAssertion:   httpz web proof linking content to a verified human credential
                   (Mina wallet signature + credential from trusted HTTPS source)
output:            Mina proof: "contentHash was created by holder of credential X"
```

**Phase 2 (AI Transparency) — future:**
```
contentHash:       Same Poseidon hash
originAssertion:   zkML proof that model M produced output O from input I,
                   where hash(O) == contentHash
output:            Mina proof: "contentHash was generated by model M"
```

**Recursive composition:**
Both branches produce proofs of the same shape (`OriginProof`). A verifier doesn't need to know which branch was used — it checks one proof type. This is the recursive composition: the zkProgram is the same, the witness (human credential vs AI inference) differs.

### On-chain state (Phase 3)

A minimal zkApp stores:
- `proofCommitment: Field` — hash of the latest verified proof
- `timestamp: UInt64` — block timestamp of submission
- `originType: Field` — 0 (human) or 1 (AI)

No content stored on-chain. The proof commitment is sufficient for verification.

## Application Design

### Creator flow

```
[Content] → [Hash] → [Choose path: Human / AI]
                          ↓                ↓
                    [httpz proof]    [zkML proof]  ← Phase 2
                          ↓                ↓
                    [OriginProof generation]
                          ↓
                    [Shareable link / QR / embed]
```

### Verifier flow

```
[Link / QR] → [Load proof from URL params or IPFS]
                   ↓
            [o1js verify() in browser]
                   ↓
            [Display: "Human-made ✓" or "AI-generated by model X ✓"]
```

### Content hashing

Content is hashed client-side using Poseidon (Mina-native hash). For large content (images, video), we hash a canonical representation:
- Text: UTF-8 bytes → Poseidon
- Images: Raw pixel bytes → chunked Poseidon (Merkle tree if needed)
- Files: Raw bytes → chunked Poseidon

The hash is the content's identity. It's what gets bound to the origin proof.

## Tech Stack

| Concern | Technology | Why |
|---------|-----------|-----|
| ZK circuits | o1js | Mina-native, TypeScript, recursive proofs |
| Content hashing | Poseidon (via o1js) | SNARK-friendly, no external dependencies |
| Human credential | httpz (Mina web proofs) | Proves data from HTTPS sources in ZK |
| AI inference proof | zkML (Mina) | Verifiable ONNX model execution — Phase 2 |
| Web framework | React + Vite | Fast dev, good o1js WASM compat |
| Styling | Tailwind CSS | Utility-first, no component library overhead |
| Package management | npm workspaces | Monorepo without extra tooling |
| Testing | vitest (web) + o1js proof simulation (circuits) | Fast, TypeScript-native |

## Phase Plan

### Phase 0: Foundation (current)
- Repo scaffold, tooling, CI
- Hello-world o1js circuit (compile, prove, verify locally)
- Basic React app shell

### Phase 1: Human Provenance
- Content hashing (Poseidon) for text input
- httpz integration: prove credential from HTTPS source
- OriginProof zkProgram: bind content hash to human credential proof
- Creator UI: paste text → generate proof → get shareable link
- Verifier UI: paste link → verify proof → see result

### Phase 2: AI Transparency
- zkML integration: prove model inference
- OriginProof Branch B: bind content hash to zkML proof
- UI: "I used AI" path with model selection

### Phase 3: On-chain
- zkApp deployment on Mina testnet then mainnet
- Proof commitment storage on-chain
- Timestamp anchoring
- Block explorer integration

### Phase 4: Distribution
- Embed code generator (iframe/widget for websites)
- QR code generation
- API for platform integrations
- Browser extension for in-page verification
